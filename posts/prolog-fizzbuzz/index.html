Prolog Learnings: FizzBuzz

December 10, 2020 3-minute read

Elixir builds on Erlang, and the first versions of Erlang were implemented in Prolog, so it shouldn’t have come as such as surprise to me that Prolog and Elixier share a number of similarities. For example, both Prolog and Elixir support argument pattern matching and using _ as an anonymous variable. However in Prolog, pattern matching behaves differently as shown below.

Here is an example of FizzBuzz as implemented in Elixir (Fizzbuzz in Elixir).

def fb(_, _, n), do: n
def fb(n), do: fb(rem(n, 3), rem(n, 5), n)

def fizzbuzz(n) do
  Enum.map(0..n, &fb/1)
end

%% Matches when Count is divisible by 15.
%% Both 3 and 5.
fizzbuzz_(0, 0, _) :-
    !,
    write('FizzBuzz'), nl.

%% Matches when Count is divisible by 5.
fizzbuzz_(0, _, _) :-
    !,
    write('Fizz'), nl.

%% Matches when Count is divisible by 3.
fizzbuzz_(_, 0, _) :-
    !,
    write('Buzz'), nl.

% Matches when Count is neither divisible by 3 or 5.
fizzbuzz_(_, _, N) :-
    !,
    write(N), nl.

fizzbuzz(Count) :-
    Count > 0,
    Fizz is Count mod 5,
    Buzz is Count mod 3,
    fizzbuzz_(Fizz, Buzz, Count),
    CC is Count - 1,
    fizzbuzz(CC).

Elixir and Prolog support pattern matching of function arguments, but;

  • Elixir performs pattern matching on an argument following any evaluation.

  • Prolog does not perform argument evaluation, pattern matching is performed on the shape of whatever is passed as the argument.

This is why in the Prolog version of FizzBuzz, the modulo calculations are first unified (Unification) with Fizz and Buzz variables which are subsequently passed as arguments to the fizzbuzz_ auxilliary (helper) function. In Elixir, the modulo is calculated and the results are passed directly as arguments to the fb helper functions.

The Elixir code looks cleaner that the equivalent Prolog, but this shouldn’t be something to become hung up on. Every language has its warts.

Here is another example - calculating the Greatest Common Denominator (GCD). In Elixir, the modulo is evaulated and the result is passed as the second argument in the recursive call to gcd.

defmodule Gcd do
  def gcd(a, 0), do: a
  def gcd(a, b), do: gcd(b, rem(a, b))
end

In Prolog, the modulo is first calculated and unified with the variable Rem and then subsequently passed as an argument to gcd. Because Prolog does not first evaulate arguments and thus attempts to match the second arguament A mod B with 0 and fails.

gcd(A, 0, A).
gcd(A, B, Result) :-
    Rem is A mod B,
    gcd(B, Rem, Result).

We should also disable backtracking using ! (a "cut") when an answer is found. Otherwise Prolog will prompt to find alternate solutions when we know that there are none (just as in the Fizzbuzz example earlier.)

gcd(A, 0, Result) :-
    !,

gcd(A, B, Result) :-
    Rem is A mod B,
    gcd(B, Rem, Result).
